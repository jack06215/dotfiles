# chezmoi run scripts: syncing content to dynamic destinations

## The problem

chezmoi is built around a simple model: a path in the source directory maps
to a path in the target (home) directory, optionally rendered through a
template. That model breaks down when the *real* destination isn't a fixed
path at all — it's generated by some other tool, at install time, and you
don't control its name. Examples:

- Browser profile directories (`~/.mozilla/firefox/xxxxxxxx.default`,
  `~/Library/Application Support/Firefox/Profiles/xxxxxxxx.default`) — the
  random prefix is assigned when the profile is created.
- IDE/editor workspace-storage folders keyed by a hash of the project path.
- Plugin/package manager cache dirs that embed a version or checksum in
  their name.
- Any path chezmoi could only know by asking another program, not by
  computing it from `.chezmoi.os`/`.chezmoi.arch`/etc.

You can't put `{{ }}` in a chezmoi source file/directory *name* and expect
it to be evaluated for regular managed entries — chezmoi does not template
target names that way. (It's an easy mistake to make: a directory literally
named `{{ .foo }}` gets created on disk with that literal string as its
name, template braces and all.) The supported escape hatch is a **script**:
its *body* is templated, and a templated shell script can compute whatever
path it wants and act on it.

This doc is a general recipe for that pattern, using the Firefox profile
case as the running example.

## The tool: `run_` scripts

A script is any source entry whose name starts with `run_`. The rest of the
name is built from optional attributes plus a free-form description:

```
run_[once_|onchange_][before_|after_]<description>.sh[.tmpl]
```

- **`run_`** — required. Marks the entry as a script rather than a file to
  place in the target tree.
- **`once_`** vs **`onchange_`** (pick at most one; omitting both means
  "run on every apply"):
  - `once_` — run the script exactly once, ever (tracked by the hash of its
    rendered contents). Reruns only if the contents change.
  - `onchange_` — run whenever the rendered contents differ from the last
    time the script ran successfully. Functionally the same mechanism as
    `once_`; the naming just communicates intent ("reruns on change" vs
    "runs once").
  - *(neither)* — no state tracking; executes on **every** `apply`/`diff`.
    Fine for cheap, idempotent commands; noisy in `diff` output otherwise.
- **`before_`** vs **`after_`** (pick at most one; omitting both runs the
  script in its normal alphabetical position among file entries):
  - `before_` — before any file/dir/symlink targets are applied this pass.
  - `after_` — after all file/dir/symlink targets have been applied this
    pass.
- **`.tmpl` suffix** — the script body is a Go template, rendered with the
  same data available everywhere else (`.chezmoi.*`, anything under
  `[data]` in `chezmoi.toml`). An empty/whitespace-only render means
  chezmoi skips running the script — the standard way to make a script
  conditional on OS/host.

## The general recipe

Say you have some managed content (a config file, a pair of files, a whole
directory) that needs to live at a destination whose exact path you can't
hardcode.

### 1. Decide how the destination path gets discovered

**Option A — precompute it once, as template data.** If you already know
the value (e.g. you copied it once from the real filesystem) or can derive
it deterministically per OS, compute it in `chezmoi.toml.tmpl` and expose it
under `[data.*]`:

```toml
{{- $base := "" -}}
{{- if eq .chezmoi.os "darwin" -}}
{{-   $base = joinPath .chezmoi.homeDir "Library/Application Support/SomeApp/Profiles" -}}
{{- else if eq .chezmoi.os "linux" -}}
{{-   $base = joinPath .chezmoi.homeDir ".someapp" -}}
{{- end -}}
{{- $id := "xxxxxxxx.default" }}
[data.someapp]
    id      = {{ $id | quote }}
    dir     = {{ joinPath $base $id | quote }}
```

Every template (including your `run_` script) can now reference
`.someapp.dir`. This is the clearest option when the value is reused in
several places, but see the [bootstrap gotcha](#a-bootstrap-gotcha) below —
it costs an extra `chezmoi apply` pass the first time.

**Option B — discover it at apply time, inside the script's own template.**
If the destination can be found with a filesystem glob (and there's
reliably exactly one match), skip `chezmoi.toml.tmpl` entirely and resolve
it inline using chezmoi's `glob` template function:

```sh
{{- $matches := glob (joinPath .chezmoi.homeDir "Library/Application Support/SomeApp/Profiles/*.default") -}}
{{- if $matches }}
#!/bin/bash
set -euo pipefail
target_dir={{ index $matches 0 | quote }}
...
{{- end }}
```

This avoids touching your own chezmoi config, so there's no bootstrap
ordering issue — but you're now trusting the glob to match exactly one
directory. Guard for zero/multiple matches if that's not guaranteed on a
fresh machine (e.g. before the app has ever been launched, the profile
directory won't exist yet).

Either way, the point is the same: push "how do we find this path" into
template evaluation, not into a source file/directory name.

### 2. Keep the canonical content as a normal managed file

Don't put the real content at the dynamic destination directly. Manage it
normally under `dot_config/` (or wherever), so it deploys to a stable,
ordinary path — this is what you actually edit, review diffs on, etc.

### 3. Write the script that projects it to the dynamic destination

A `run_onchange_after_*.sh.tmpl` that creates the destination directory and
symlinks the managed files into it:

```sh
{{- if eq .chezmoi.os "darwin" -}}
#!/bin/bash
set -euo pipefail

target_dir={{ .someapp.dir | quote }}
source_dir={{ joinPath .xdg.configHome "someapp/profile" | quote }}

mkdir -p "$target_dir"
ln -sf "$source_dir/config.json" "$target_dir/config.json"
{{- end }}
```

Why `after_`: guarantees the managed files under `source_dir` already exist
on disk (chezmoi applies regular file entries before `run_after_` scripts).
Why `onchange_`: the script only reruns — and only shows up in `chezmoi
diff` — when the rendered path or logic actually changes, not on every
apply. Why symlink rather than copy: edits to the managed source take
effect immediately, with nothing to keep back in sync.

### 4. Guard by OS/host

Wrap the whole script body in `{{ if ... }} ... {{ end }}` for whatever
condition applies (`.chezmoi.os`, `.myComputer.hostname`, etc.). An
empty render means chezmoi skips executing the script — cleaner than a
runtime `exit 0` inside a shell conditional.

### 5. Verify

```sh
chezmoi apply --dry-run --verbose   # see the rendered script + any diff
chezmoi apply --verbose             # actually run it
chezmoi state dump | jq '.scriptState'   # confirm it registered as run
chezmoi apply --dry-run --verbose   # rerun; should now be a no-op
```

If the first `--dry-run` shows a `map has no entry for key "..."` template
error and you took Option A, see the bootstrap gotcha below — you likely
need to `chezmoi apply` once first so the new data field actually lands in
`~/.config/chezmoi/chezmoi.toml`.

## Execution order in one `apply` pass

```
run_before_* scripts   (alphabetical among themselves)
        ↓
all file/dir/symlink targets   (alphabetical, dot_config/, AppData/, etc.)
        ↓
run_after_* scripts    (alphabetical among themselves)
```

## How `once_`/`onchange_` tracking actually works

chezmoi keeps a small per-machine, per-user state database, separate from
the source repo (not version-controlled):

```
~/.config/chezmoi/chezmoistate.boltdb
```

For each `once_`/`onchange_` script, chezmoi:

1. Renders the script's template to get its final text.
2. SHA-256 hashes that rendered text.
3. Looks the hash up in the `scriptState` bucket of the state db.
   - **Not found** → run the script, then record `{name, runAt: <timestamp>}`
     under that hash.
   - **Found** → skip; nothing is executed and nothing shows up in `diff`.

Inspect it directly:

```sh
chezmoi state dump | jq '.scriptState'
```

Because the key is the hash of the *rendered script*, not of any files it
touches or creates, the script only reruns when its own template output
changes — a literal edit, or a referenced variable resolving to a new
value. If whatever it created is later modified or deleted out of band (by
another program, manually, ...), chezmoi won't notice or self-heal it — the
hash hasn't changed. Use a plain `run_` (no `once_`/`onchange_`) instead if
you want a script to unconditionally re-assert state on every apply, at the
cost of it always showing up in `diff`.

## A bootstrap gotcha

If you took Option A above and your own `chezmoi.toml.tmpl` is itself a
chezmoi-managed dotfile (i.e. it lives under `dot_config/chezmoi/` and
targets `~/.config/chezmoi/chezmoi.toml`, rather than the special
root-level `.chezmoi.toml.tmpl` chezmoi reads before loading config),
templates in an `apply` run are evaluated using whatever config is
**already on disk** at the start of that run — not the one being
regenerated mid-run. So adding a new field under `[data.*]` requires
**two** `chezmoi apply` passes to become usable elsewhere:

1. First pass updates `~/.config/chezmoi/chezmoi.toml` with the new field
   (any template referencing the new field elsewhere fails during this
   pass with a `map has no entry for key "..."` error).
2. Second pass loads the updated config, and the new field is available
   everywhere, including in scripts.

This doesn't apply if you use Option B (inline `glob`/`output` discovery),
since there's no dependency on your own config being regenerated first.

## Worked example: Firefox profile symlinks

Applying the recipe above to Firefox profiles (Option A, since the profile
ID is stable and reused elsewhere):

`dot_config/chezmoi/chezmoi.toml.tmpl` computes the real path once, per OS:

```toml
[data.firefox]
    profile      = "vfop1w54.default"
    profileDir   = "/Users/jack.cho/Library/Application Support/Firefox/Profiles/vfop1w54.default"
```

`dot_config/firefox/profile/{user.js,chrome/userChrome.css}` are managed
normally and deploy to `~/.config/firefox/profile/` — the single source of
truth, editable like any other dotfile.

`run_onchange_after_link-firefox-profile.sh.tmpl` runs last, only on
darwin, and only when its rendered content changes:

```sh
{{- if eq .chezmoi.os "darwin" -}}
#!/bin/bash
set -euo pipefail

profile_dir={{ .firefox.profileDir | quote }}
source_dir={{ joinPath .xdg.configHome "firefox/profile" | quote }}

mkdir -p "$profile_dir/chrome"
ln -sf "$source_dir/user.js" "$profile_dir/user.js"
ln -sf "$source_dir/chrome/userChrome.css" "$profile_dir/chrome/userChrome.css"
{{- end }}
```

Net result: edit `dot_config/firefox/profile/user.js`, run `chezmoi apply`,
and the real Firefox profile picks up the change through the symlink —
without chezmoi ever needing to know the profile ID as part of a path it
manages directly.
